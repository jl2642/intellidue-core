# Security Policy

## Supported versions

Security fixes are applied to the current `main` branch and the latest published `core-vX.Y.Z` release, once a formal release exists. Older development snapshots are not supported.

## Reporting a vulnerability

Do not open a public issue for an undisclosed vulnerability, credential, private project identifier or data-exposure concern.

Use GitHub's private **Report a vulnerability** channel from the repository Security tab when available. If that channel is unavailable, contact the repository owner through the GitHub profile before sharing technical details.

A useful report includes:

- affected commit or release;
- reproducible steps using synthetic data only;
- impact and attack prerequisites;
- proof that no real project data was accessed or copied;
- a proposed mitigation, when available.

## Data boundary

This public repository must never contain real project sources, filenames, hashes, workpapers, Reader products, commercial facts, legal findings, signatures, tokens or private cloud links. Report suspected boundary violations privately and preserve the evidence without redistributing it.

## Coordinated handling

The maintainer will acknowledge a complete report, assess severity, prepare a private fix, validate the release and publish an advisory when disclosure is appropriate. No service-level response time is promised by this non-commercial repository.
