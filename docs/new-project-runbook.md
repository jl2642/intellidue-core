# New Private Project Runbook

## A. Project-owner preparation

Before analysis begins, the project owner should provide:

1. a unique project ID and project name;
2. the intended decision, audience, deadline and confidentiality level;
3. a private source room or uploaded source packages;
4. known commercial context, prior conclusions and unresolved concerns;
5. the requested product class: screening memo, preliminary/red-flag DD, conditional specialist DD, integrated DD, transaction-protection report or decision memo;
6. named professional reviewers when the output will be formally relied upon.

The project owner should not reorganize or rename the original source room solely for AI convenience. The original folder structure should be preserved as evidence, with a controlled staged copy used for processing.

## B. ChatGPT project bootstrap

The authorized ChatGPT workspace should:

1. create the immutable project identity and intake timestamp;
2. inventory every file and compute duplicate identities;
3. record source disposition, review depth and exclusions;
4. distinguish physical-file count, unique-source count, reviewed key sources and extracted information units;
5. create the Question Map, Evidence Register, conflict register, calculation register, open-request list, critical gates and blocked-output list;
6. confirm whether the available evidence supports only a data-gap notice, preliminary issue report, conditional DD or a stronger product;
7. establish Reader and Control package boundaries before writing management products.

No report may claim that every registered file was professionally reviewed unless the workpapers actually evidence that procedure.

## C. Professional production sequence

The default sequence is:

1. high-materiality workpapers and recalculations;
2. specialist reports supported by those workpapers;
3. integrated report assembled from accepted specialist parents;
4. management summary or stage-gate memo derived from the integrated report;
5. transaction-protection, evidence-closure or bespoke follow-up products as required;
6. Reader/Control separation, publication QA and clean-chat portability acceptance.

Each material conclusion should identify its source or workpaper authority, contrary evidence, remaining gap, transaction or decision effect and next closing action.

## D. Private-runtime promotion

For an accepted project release:

1. prepare a project-local `adapter.json` binding one `project_id`, the three control contracts and non-empty Reader/Control roots;
2. validate the private project with IntelliDue Core;
3. build and validate a deterministic private release package;
4. promote into a clean project-bound runtime;
5. verify Current, Last-success and Archive pointers;
6. retain the previous immutable release;
7. test rollback and recovery when the release changes a material product or control contract.

The private runtime packages already-controlled products. It does not generate or analyze the reports.

## E. Incremental evidence

When new evidence arrives:

1. register only new or changed files;
2. update the impact map from source to Evidence, Question, workpaper and report;
3. regenerate only affected products;
4. retain unresolved conflicts and prior versions;
5. promote a restricted output only when its explicit evidence gate passes;
6. archive the previous Current release and record the new Last-success pointer.

## F. Finalization

A project may be called Final Full DD only when:

- all material evidence gates required by the stated scope are closed or explicitly accepted by the decision authority;
- required external confirmations, searches, fieldwork, testing and professional procedures are complete;
- finance, legal, technical/HSE and other applicable reviewers have signed the relevant products;
- valuation, net debt, closing funds and investment-return outputs are supported by controlled inputs;
- the integrated report and management memo remain consistent with the professional parents;
- Reader, Control, package integrity and clean-chat acceptance all pass.
